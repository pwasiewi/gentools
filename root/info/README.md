# Gentoo UEFI Secure Boot — Script Documentation

### go to [README.txt](root/info/gentonubuntu.txt) for exemplary installation in virtualbox

## Overview

Two scripts covering a complete UEFI Secure Boot setup for Gentoo.
Each boot method can be installed independently and in any order.

```
efi-keys     — key generation + optional UEFI enrollment
efi-boot     — boot method installer (GRUB / shim / EFI Stub)
```

### Packages

| Tool | Package (Gentoo) | Purpose |
|-----------|----------------|----------|
| `openssl` | dev-libs/openssl | RSA/X.509 key generation |
| `sbsign`, `sbverify` | app-crypt/sbsigntools | EFI signing and verification |
| `cert-to-efi-sig-list`, `sign-efi-sig-list`, `efi-readvar`, `efi-updatevar` | app-crypt/efitools | certificate conversion, AUTH files, UEFI variable read/write |

| `/usr/share/shim/BOOTX64.EFI` | sys-boot/shim-signed | shim bootloader (shim method) |

### Environment

- System booted via UEFI (not BIOS/CSM)
- ESP mounted — default `/boot/efi`
- Root privileges

## efi-keys — Key Generation

### What it does

Generates four keys forming the Secure Boot hierarchy:

```
PK  (Platform Key)        — root of trust, enrolled by platform owner
KEK (Key Exchange Key)    — signs changes to db/dbx
db  (Signature Database)  — keys of accepted EFI signatures
MOK (Machine Owner Key)   — user key, verified by shim
```

Preserves existing Microsoft keys (read from UEFI via `efi-readvar`)
and appends them to KEK.esl and db.esl — Windows remains functional.

Exports `*.auth` files to ESP in the `EFIKEYS/` directory, accessible
from EFI Shell as `FS0:\EFIKEYS\`.

### Usage

```bash
# Default — back up UEFI keys, then generate new ones
sudo efi-keys

# Backup only — does not generate new keys
sudo efi-keys --backup-only

# Backup + generation + create restore AUTH files
sudo efi-keys --make-restore-auth

# Backup + attempt automatic UEFI enrollment (requires Setup Mode)
sudo efi-keys --enroll

# Without backup (not recommended)
sudo efi-keys --no-backup

# With custom working directory and ESP
sudo efi-keys --workdir /root/secureboot --esp /boot/efi
```

### Options

| Option | Default | Description |
|-------|-----------|------|
| `--workdir DIR` | `~/secureboot` | working directory for keys and backups |
| `--esp DIR` | `/boot/efi` | ESP mount point |
| `--enroll` | off | attempt `efi-updatevar` after key generation |
| `--backup-only` | off | backup only, no key generation |
| `--no-backup` | off | skip backup (not recommended before BIOS operations) |
| `--make-restore-auth` | off | after key generation create restore_*.auth from backup ESLs |

### UEFI Key Backup

Backup runs **automatically as the first operation** on every invocation
(unless `--no-backup` is given). Always back up BEFORE any BIOS key management operations.

```
~/secureboot/backup/
├── 20260101-120000/     ← timestamped directory
│   ├── PK.esl           ← Platform Key
│   ├── KEK.esl          ← Key Exchange Keys (MS + Lenovo)
│   ├── db.esl           ← Signature Database
│   └── dbx.esl          ← Forbidden Signatures
└── latest -> 20260101-120000   ← symlink to latest backup

/boot/efi/EFIKEYS-backup/
└── 20260101-120000/     ← copy on ESP (accessible from EFI Shell)
```

Backup ESL files are automatically reused during key generation
— MS/Lenovo entries are appended to `KEK.esl` and `db.esl` without querying UEFI again.

**Backup limitation:** ESL files are not standalone re-enrollable through BIOS
without being wrapped in signed AUTH files. Restore requires Setup Mode via `efi-updatevar`
or a BIOS Key Management that accepts raw CER/ESL uploads.

### restore_*.auth Files — Restoring MS/Lenovo Key Content

The `--make-restore-auth` option creates AUTH files from original backup ESLs,
signed with the **own** KEK/PK generated by the script.

```
~/secureboot/backup/restore-auth/
├── restore_db.auth    ← original MS/Lenovo db, signed with own KEK
├── restore_KEK.auth   ← original MS/Lenovo KEK, signed with own PK
└── restore_dbx.auth   ← original dbx, signed with own KEK

/boot/efi/EFIKEYS-restore/   ← copy on ESP
```

Restore when own keys are active in UEFI:
```bash
efi-updatevar -f ~/secureboot/backup/restore-auth/restore_db.auth  db
efi-updatevar -f ~/secureboot/backup/restore-auth/restore_KEK.auth KEK
efi-updatevar -f ~/secureboot/backup/restore-auth/restore_dbx.auth dbx
```

**Key limitation:** `restore_*.auth` restore the **content** (MS/Lenovo keys)
but are authenticated by your key — they are not original MS-signed files.
They only work while your own PK is active in UEFI.
To return to factory MS state use: `BIOS → Restore Factory Keys`.

### Generated Files

```
~/secureboot/keys/
├── PK.key / PK.crt / PK.cer / PK.esl / PK.auth
├── KEK.key / KEK.crt / KEK.cer / KEK.esl / KEK.auth
├── db.key / db.crt / db.cer / db.esl / db.auth
├── MOK.key / MOK.crt / MOK.der
├── old_KEK.esl / old_db.esl          ← MS keys preserved from UEFI
└── dbx.auth                           ← if dbx was present in UEFI

/boot/efi/EFIKEYS/
├── PK.auth / KEK.auth / db.auth       ← for BIOS enrollment
├── PK.cer / KEK.cer / db.cer / MOK.der
└── dbx.auth                           ← if present
```

Keys are generated once. Re-running with an existing `db.key` exits without
overwriting — delete the `keys/` directory to force regeneration.

### Enrollment — `--enroll` option

Requires UEFI in **Setup Mode** (PK cleared in BIOS).
The script reads `SetupMode` from `/sys/firmware/efi/efivars/` and warns
if the mode is incorrect. Enrollment order is critical:

```
db → KEK → PK
```

PK enrolled last — writing it activates User Mode and from that point
UEFI verifies signatures of all EFI files.

### Manual BIOS Enrollment

If `--enroll` fails (most consumer boards block
`efi-updatevar` even in Setup Mode):

1. Enter BIOS → Secure Boot → Key Management → **Custom Mode**
2. Clear existing keys (Reset to Setup Mode)
3. Enroll in order: `db.auth` → `KEK.auth` → `PK.auth`
4. Files available on ESP or via EFI Shell: `FS0:\EFIKEYS\`

## efi-boot — Boot Method Installer

### What it does

Installs one or all EFI boot methods using keys generated by `efi-keys`.
Each method is independent — they can be installed separately and in any order.

### Usage

```bash
# All methods
sudo efi-boot

# Specific method only
sudo efi-boot --method stub-signed
sudo efi-boot --method stub-luks --luks-uuid <UUID>
sudo efi-boot --method grub
sudo efi-boot --method shim

# With parameters
sudo efi-boot \
    --method stub-signed \
    --disk /dev/nvme0n1 \
    --part 1 \
    --kernel /boot/kernel-genkernel-x86_64-6.14.6-zen1 \
    --cmdline "ro quiet splash"

# With LUKS — UUID from: cryptsetup luksUUID /dev/nvme0n1p3
sudo efi-boot \
    --method stub-luks \
    --luks-uuid d3874106-3000-43a7-9624-bcf283d0fcf6
```

### Options

| Option | Default | Description |
|-------|-----------|------|
| `--method METHOD` | `all` | boot method |
| `--disk DEV` | `/dev/nvme0n1` | disk for efibootmgr |
| `--part N` | `1` | ESP partition number |
| `--esp DIR` | `/boot/efi` | ESP mount point |
| `--kernel PATH` | auto | kernel path |
| `--initrd PATH` | auto | initrd path |
| `--cmdline STRING` | `ro quiet splash` | additional kernel parameters |
| `--workdir DIR` | `~/secureboot` | must match path used in efi-keys |
| `--luks-uuid UUID` | auto | LUKS2 UUID for stub-luks method |

## Boot Methods — Security Ranking

From most to least secure when Secure Boot keys are wiped from UEFI.

| Rank | Method | Trust chain | Key wipe |
|------|--------|-----------------|-------------------|
| 1 | `stub-signed` ✓ | UEFI → own db → kernel | nothing boots — full lockout |
| 1 | `stub-luks` ✓ | UEFI → own db → kernel → LUKS2 | nothing boots — full lockout + encryption |
| 1 | `stub` | UEFI → own db → kernel (embedded initrd) | same as above |
| 2 | `grub` | UEFI → own db → GRUB → kernel | nothing boots |
| 3 | `shim` | UEFI → MS PK → MS db → shim → MOK → GRUB → kernel | Gentoo blocked, Windows works |
| 4 | `stub-signed` + db append | UEFI → MS PK → MS KEK → db (MS+own) → kernel | kernel does not boot after entry removal |

Ranks 1-2 require own PK enrolled in UEFI (Setup Mode).
Rank 3 does not require Setup Mode — MOK enrolled from running OS.
Rank 4 does not require Setup Mode — db.cer appended via BIOS Key Management.

### `grub` — GRUB Standalone Signed with db Key `[rank 2]`

**When to use:** you have own PK/KEK/db enrolled in BIOS (via efi-keys).
No shim or MOK needed. Simplest setup when you have full control over UEFI keys.

**How it works:** `grub-mkstandalone` builds a standalone `grubx64.efi`
with embedded configuration from `/boot/grub/grub.cfg`. The file is signed
with the `db` key and verified directly by UEFI firmware.

**Files on ESP:**
```
/boot/efi/EFI/Gentoo/grubx64.efi
```

**UEFI entry:** `Gentoo Secure Boot (GRUB)`

**After grub.cfg update:** rebuild required (re-run `--method grub`).

---

### `shim` — GRUB Signed with MOK + Shim as Intermediary `[rank 3]`

**When to use:** you want to keep original Microsoft keys in BIOS
(e.g. dual-boot with Windows without enrolling own PK). Shim is signed
with MS key and verified by firmware, then shim verifies GRUB with MOK.

**How it works:**
```
UEFI firmware → shimx64.efi (MS-signed) → grubx64.efi (MOK-signed) → kernel
```

MOK must be registered via `mokutil` or MokManager on first boot.

**Files on ESP:**
```
/boot/efi/EFI/GentooShim/
├── shimx64.efi    ← from /usr/share/shim/BOOTX64.EFI
├── mmx64.efi      ← MokManager
├── grubx64.efi    ← MOK-signed
└── MOK.der        ← certificate for import
```

**UEFI entry:** `Gentoo Shim Boot`

**Required action after install:**
```bash
mokutil --import /boot/efi/EFI/GentooShim/MOK.der
# reboot → MokManager → Enroll MOK
```

---

### `stub` — EFI Stub with initrd and cmdline Embedded in Kernel `[rank 1]`

**When to use:** you want to boot without a bootloader, with external initrd
embedded in the EFI file via `objcopy`. One signed file contains kernel, initrd and cmdline.

**How it works:** `objcopy` embeds initrd as `.initrd` section and cmdline
as `.cmdline` section in the EFI kernel. Firmware loads one file directly.

**Requirements:** kernel with EFI Stub support (`CONFIG_EFI_STUB=y`), external initrd.

**Files on ESP:**
```
/boot/efi/EFI/GentooEFISTUB/vmlinuz-signed.efi
```

**UEFI entry:** `Gentoo EFI Stub (initrd embedded)`

**After kernel or initrd update:** rebuild required.

---

### `stub-signed` — EFI Stub with Built-in initramfs ✓ Confirmed `[rank 1]`

**When to use:** kernel compiled with `CONFIG_INITRAMFS_SOURCE` pointing
to a pre-built initramfs — no external initrd needed. Simplest
and fastest EFI Stub variant. **Confirmed working.**

**How it works:** kernel is only signed with the `db` key and copied to ESP.
Cmdline passed via `efibootmgr --unicode` (stored in NVRAM).

**Files on ESP:**
```
/boot/efi/EFI/GentooEFISTUB/vmlinuz-signed.efi
```

**UEFI entry:** `Gentoo EFI Stub`

**After kernel update:** just re-run `--method stub-signed`.

---

### `stub-luks` — EFI Stub with LUKS2 Encryption (genkernel) `[rank 1]`

**When to use:** root on a LUKS2 partition, genkernel built with `--luks`.
Equivalent to `stub-signed` but with decryption parameters in NVRAM.

**How it works:** kernel signed with `db` key, cmdline contains `crypt_root=UUID=`
and `root=/dev/mapper/gentoo_crypt` passed via `efibootmgr --unicode`.
Kernel has `CONFIG_CMDLINE` without `OVERRIDE` — efibootmgr appends to built-in cmdline.

```
UEFI → own db → signed kernel → genkernel initramfs → cryptsetup luksOpen → btrfs/ext4 root
```

**Requirements:**
```bash
# genkernel with LUKS support
genkernel --luks initramfs

# /etc/conf.d/dmcrypt
target=gentoo_crypt
source=UUID=<LUKS_UUID>
```

**Files on ESP:**
```
/boot/efi/EFI/GentooEFISTUB/vmlinuz-signed-luks.efi
```

**UEFI entry:** `Gentoo EFI Stub LUKS`

**Installation:**
```bash
# UUID from: cryptsetup luksUUID /dev/nvme0n1p3
sudo efi-boot --method stub-luks \
    --luks-uuid d3874106-3000-43a7-9624-bcf283d0fcf6
```

**genkernel vs dracut note:** genkernel uses `crypt_root=UUID=` not `rd.luks.uuid=`.
The script uses genkernel syntax. If using dracut — change in `setup_efistub_luks()`.

---

### Rank 4 Variant — db Append Without Own PK (e.g. ThinkPad Deployed Mode)

**When to use:** BIOS does not support Setup Mode or is in Deployed Mode
(ThinkPad T14 Gen 6 and newer from 2025). You cannot or do not want to replace
Lenovo/MS PK/KEK.

**How it works:** own `db.cer` appended to existing MS database via
BIOS Key Management — without knowing KEK.key. Kernel signed
with own `db.key`. UEFI accepts it alongside MS signatures.

**BIOS steps:**
```
Security → Secure Boot → Key Management
         → Authorized Signatures (db) → Enroll Signature
         → select: /boot/efi/EFIKEYS/db.cer
```

Then normal signing:
```bash
sudo efi-boot --method stub-signed
```

**Limitations:** after resetting db to MS defaults your entry disappears
and the kernel stops booting. Weaker isolation — relies on MS infrastructure.
Does not protect against replacement of MS-signed bootloaders.

## Typical Workflows

### Before Any BIOS Operation — Back Up Keys

```bash
# Always run before entering BIOS Key Management
sudo efi-keys --backup-only
# Backup lands in ~/secureboot/backup/latest/ and on ESP/EFIKEYS-backup/
```

### First Run — Own Keys

```bash
# 1. Generate keys
sudo efi-keys

# 2. Enroll keys in BIOS (Setup Mode → Custom Mode → enroll AUTH)

# 3. Install chosen methods
sudo efi-boot --method stub-signed   # works immediately
sudo efi-boot --method grub          # optional
```

### First Run — Keeping MS Keys (dual-boot)

```bash
sudo efi-keys         # MOK generated automatically
sudo efi-boot --method shim
# reboot → MokManager → Enroll MOK
```

### Installation with LUKS2 Encryption

```bash
# 1. Prepare LUKS2 partition and unpack livecd
sudo montuj --fs btrfs-ssd --luks /dev/nvme0n1p3
# montuj prints LUKS UUID at the end

# 2. Enter chroot and build genkernel with --luks
genkernel --luks initramfs

# 3. Sign kernel and register UEFI entry with LUKS cmdline
sudo efi-boot --method stub-luks \
    --luks-uuid <UUID-from-montuj>

# 4. Add to /etc/conf.d/dmcrypt:
# target=gentoo_crypt
# source=UUID=<LUKS_UUID>
rc-update add dmcrypt boot
```

### Kernel Update

```bash
# Sign only and update entry — no keys are regenerated
sudo efi-boot --method stub-signed --kernel /boot/kernel-new
```

### grub.cfg Update

```bash
# Rebuild GRUB with new grub.cfg
sudo efi-boot --method grub
```

### After Windows Overwrites the Bootloader

```bash
cp /boot/efi/EFI/Microsoft/Boot/bootmgfw.efi /boot/efi/EFI/Boot/bootx64.efi
# or restore Gentoo entry via efibootmgr:
efibootmgr --create --disk /dev/nvme0n1 --part 1 \
    --label "Gentoo EFI Stub" \
    --loader '\EFI\GentooEFISTUB\vmlinuz-signed.efi' \
    --unicode "root=UUID=<UUID> ro quiet splash"
```

## Limitations and Known Issues

**`efi-updatevar` on consumer boards** — most firmware blocks
EFI variable writes even in Setup Mode. The `--enroll` option is best-effort.
Works more reliably on server boards and ThinkPads.

**BIOS key enrollment order** — always db → KEK → PK.
PK last activates User Mode. Reversed order may block
enrollment of remaining keys.

**`stub` vs `stub-signed`** — `stub` with `objcopy` requires precise
PE section alignment (VMA). On some kernels values `0x20000`
and `0x3000000` may conflict with existing sections — symptom is kernel
panic or missing initrd at boot. `stub-signed` does not have this problem.

**Rebuild after update** — no method is automatically updated after `emerge @world`.
Manual call to `efi-boot` with the appropriate method is required.
Can be added to `/etc/kernel/postinst.d/`.

**Private keys** — `*.key` files should not reside on the ESP.
Keep `~/secureboot/keys/` outside the filesystem accessible from UEFI.

## Other Scripts

### build-usb — Gentoo Live USB Drive

Creates a bootable Gentoo Live USB: partitions (GPT: ESP 512MiB + data partitions),
formats, installs GRUB with `--removable --no-nvram` and generates `grub.cfg`.

Partition layout grows as a triangular series to fill the drive:
ESP(512MiB) + LIVE(≤30GiB) + DATA3(≤40GiB) + DATA4(≤60GiB) + DATA5(≤90GiB) + …
Each cap increases by 10 GiB more than the previous step.

```bash
sudo build-usb /dev/sdX
```

Safety: refuses to operate on disks with `removable=0` or `ID_BUS!=usb`.
After running, manually copy kernel, initrd and squashfs:

```bash
mount /dev/sdX1 /mnt/efi
cp gentoo initrd.uuid.cpio.gz /mnt/efi/
mount /dev/sdX2 /mnt/data
cp livecd.squashfs /mnt/data/
```

### build-nvme — NVMe Disk Triple-Boot

Partitions NVMe for triple-boot Gentoo + Ubuntu + Windows 11, formats Linux
partitions, installs GRUB, generates `grub.cfg` and `fstab.gentoo` with real UUIDs.

```bash
sudo build-nvme /dev/nvme0n1
```

Layout: ESP(1G) + MSR(16M) + GENTOO(60G) + UBUNTU(60G) + SWAP(16G) + HOME(500G) + WINDOWS(rest).
MSR and WINDOWS are left unformatted — Windows installer handles them automatically.

Safety: refuses to operate on removable devices (`removable=1`).

---

### build-initrd-uuid-next — initrd for Gentoo Live USB

Builds a minimal initrd (static BusyBox) that locates the livecd partition by UUID,
mounts squashfs via loop and bind-mounts directories to RAM before `switch_root`.

```bash
build-initrd-uuid-next
build-initrd-uuid-next --force-rebuild-busybox
```

Key features: BusyBox binary cache (skips compilation on subsequent builds),
`/mnt/cdrom` survives `switch_root` (bind mount inside ROOT\_MNT), verification
via `/proc/mounts` instead of `mountpoint -q` (BusyBox compatibility).

### montuj — Gentoo Installation onto Partition

Formats a partition with the chosen FS profile, optionally creates a LUKS2 container,
mounts and unpacks `livecd.squashfs` (replaces `montuj-btrfs`, `montuj-ssd`,
`montuj-evo` and `msquash`).

```bash
# btrfs on SSD (default)
sudo montuj /dev/nvme0n1p3

# ext4 with SSD tuning
sudo montuj --fs ext4-ssd /dev/nvme0n1p3

# btrfs + LUKS2
sudo montuj --fs btrfs-ssd --luks /dev/nvme0n1p3
```

FS profiles: `btrfs-ssd` (default), `btrfs`, `ext4-ssd`, `ext4-evo`, `ext4`.

squashfs must be available at `/mnt/cdrom/livecd.squashfs` — ensured by the new initrd
via bind mount of the livecd partition inside ROOT\_MNT before `switch_root`.

### v — Gentoo LiveUSB Generation

Main script for building a Gentoo LiveUSB. Manages chroot, kernel, squashfs
and ISO layout. Operates on profiles defined in `/etc/liveusb/profiles/`.

```bash
v <profile> <command>
```

| Command | Description |
|---------|-------------|
| `e` | Enter chroot (mounts /dev, /proc, /sys from host) |
| `eb` | Enter already-mounted chroot from a second console |
| `u` | Unmount /dev, /proc, /sys from chroot |
| `k` | Build kernel via genkernel for live USB |
| `s` | Compress system into `livecd.squashfs` |
| `a` | `cleanall` + `s` + `i` |
| `i` | Populate ISO directory layout (isolinux files) |
| `f` | Download and extract Gentoo stage3 |
| `0`–`8` | Run install stage N inside chroot |
| `0host`–`8host` | Run stage N directly on host |
| `cfg` | Fetch gentools repository config |
| `p` / `r` | Open profile/chroot dir in mc |

### e — Emerge Portage Wrapper

Single-letter shortcuts for common `emerge` operations.


### ve — Portage Config File Editor

Validates and appends entries to Portage config files with syntax checking.
Shorthand aliases: `veu` (use), `vex` (keywords), `ves`/`vis` (mask),
`vec`/`vic` (make.conf), `vie` (env), `vip`/`vep` (provided).

```bash
ve kw "=dev-libs/foo-1.2.3 **"      # package.accept_keywords
ve use "=dev-libs/foo-1.2 bar -baz"  # package.use
ve mask "=dev-libs/foo-1.2"          # package.mask
ve env "=dev-libs/foo-1.2 debug.conf" # package.env
ve provided "=dev-libs/foo-1.2.3"    # package.provided
ve conf 'MAKEOPTS="-j32"'            # make.conf
ve kw                                 # open vim on keywords file
```

Atom format is validated before writing: `[operator]category/package[-version]`.

### luks — LUKS2 Disk Management

Unified tool for opening, closing, creating and managing LUKS2 partitions.

```bash
luks open   /dev/sda2 data             # open + mount at /mnt/data
luks open   /dev/sda2 data -k /etc/luks/key.key   # with keyfile
luks open   /dev/sda2 data --compress=zstd         # btrfs + compression
luks close  /dev/sda2                  # unmount + close mapping

luks create /dev/sda2                  # LUKS2 + ext4
luks create /dev/sda2 -t btrfs -L mydata   # btrfs with label
luks create /dev/sda2 -k /etc/luks/key.key  # keyfile encryption

luks passwd /dev/sda2 add             # add passphrase
luks passwd /dev/sda2 remove          # remove passphrase
luks passwd /dev/sda2 change          # change passphrase
```

### 8z — Archive and Compression

Multi-mode compression using 7z/zstd, optimized for many-core CPUs (32 threads).

```bash
8z backup /etc                  # auto-detect, → backup.tar.7z
8z -f videos /mnt/video         # fast: zstd -T32 -6  (~200 MB/s)
8z -n data /var/log             # normal: LZMA2 mx6   (~50 MB/s)
8z -m src /home/pwas/code       # max: LZMA2 mx9      (~15 MB/s)
8z -u src /home/pwas/code       # ultra: LZMA2 mx9 1536m (~10 MB/s, ~5GB RAM)
8z -x backup.tar.7z             # extract
8z -x archive.tar.gz            # extract any supported format
```

Auto mode detects content type and selects the appropriate compression level.
Extraction supports: `.tar.7z`, `.7z`, `.tar.zst`, `.tar.gz`, `.tar.bz2`, `.tar.xz`, `.zip`, `.rar`.

### vidconv — Video Conversion

Converts video files using ffmpeg with codec and quality selection.

```bash
vidconv film.mov                            # → film.mp4 (h265, crf28, aac)
vidconv -c h264 -q 20 film.mts             # h264, high quality
vidconv -c h265 -p slow -q 24 film.mp4     # better h265, slower
vidconv -hw -c h264 film.mov               # NVIDIA NVENC (GPU)
vidconv -f mkv -a flac film.mts            # MKV + FLAC audio
vidconv -c copy -f mp4 film.mov            # remux only, no re-encode
vidconv -s 1080 *.mp4                      # batch scale to 1080p
vidconv -2 -c h265                         # batch all video in current dir
vidconv -c av1 -f webm film.mp4            # AV1/WebM for streaming
```

Codecs: `h265` (default), `h264`, `av1`, `vp9`, `copy`.
Formats: `mp4` (default), `mkv`, `webm`, `avi`.
Input: `.mov .mp4 .mts .m2ts .mkv .avi .wmv .3gp .webm .flv .ts .vob .mpg .hevc .dav`.

### watermark2jpgsep — Photo Processing Pipeline

Generates thumbnail variants (BW/grayscale/sepia) with steganographic watermark (steghide).

```bash

```

Output variants per image: `_bw` (monochrome), `_gr` (grayscale), `_sp` (sepia from _gr),
`_ms` (sepia from _bw). All prefixed with `min`.
